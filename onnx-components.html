What ONNX actually is

A trained PyTorch model is a graph of operations expressed in PyTorch's Python API. You cannot run that graph directly inside a browser or in a generic C++ runtime without dragging PyTorch along. ONNX, the Open Neural Network Exchange, is an intermediate representation for that graph. It stores the same math as a list of standardized operators plus their initial weights, serialized into a binary file.

An ONNX file is a serialized protobuf message. It holds three things: a list of nodes (each node is one operator like MatMul, Add, or Softmax), a list of initializers (the trained weight tensors), and metadata describing inputs, outputs, and the opset version. The opset is the dialect of operators the graph uses. This bundle targets opset 17, which onnxruntime-web 1.21 and later supports in the browser.

A separate program called a runtime loads the protobuf and executes the operator graph. onnxruntime is the reference runtime from Microsoft. It ships builds for Python, C++, and JavaScript. The JavaScript build, onnxruntime-web, can run the same file through a WASM CPU provider or a WebGPU provider. That portability is the whole point. You export once, then run the file anywhere a runtime exists.

Export is the act of turning a PyTorch module into an ONNX file. PyTorch's exporter walks the module with example inputs, records every tensor operation into the ONNX operator vocabulary, and writes the protobuf. This process is called tracing. The example inputs you feed during tracing fix the operations recorded, so you must declare which input dimensions are allowed to vary later through dynamic axes.

The export pipeline this article describes lives in the indictrans2-onnx-export repository. It exports the AI4Bharat IndicTrans2 translation models into browser-ready ONNX bundles. Each bundle ships to a Hugging Face repo and is consumed by a browser voice chat app. Every component below comes from that real pipeline.

The model you start with

Encoder, decoder, and autoregression

IndicTrans2 is a sequence-to-sequence transformer. It translates a source sentence into a target sentence. The model has two halves. The encoder reads the source tokens and produces a contextual embedding for each one. The decoder generates target tokens one at a time, attending back to the encoder outputs at every step.

Generation is autoregressive. The decoder produces token 1, then token 2 conditioned on token 1, then token 3 conditioned on tokens 1 and 2, and so on until it emits an end of sentence token. Each new token depends on every previously generated token. This dependency is what makes the export split into multiple graphs.

IndicTrans2 architecture constants

The export scripts hardcode a few constants that must match the model's config.json. For the en to indic 200M model, the encoder and decoder each have 18 layers. Each layer has 8 attention heads. The embedding dimension is 512, so each head has dimension 64. These numbers drive the shape of every past and present KV tensor in the exported graphs.

The decoder starts from token id 2, the decoder_start_token_id, and stops when it emits token id 2, the eos_token_id. The pad token is id 1, the bos token is id 0, and the unknown token is id 3. The source vocabulary holds 32322 entries. The target vocabulary holds 122672 entries.

Why one model becomes three graphs

The KV cache problem

Naive autoregressive decoding recomputes the full decoder forward pass for every new token. To generate token N, the decoder reprocesses tokens 1 through N minus 1. That is quadratic in the output length and wasteful, because the attention keys and values for earlier tokens do not change when a new token is added.

The fix is the KV cache. At each step the decoder computes the key and value tensors for the current token and appends them to a running cache. The next step receives that cache as input and only computes attention for the new token against the cached history. This makes generation linear in the output length.

The cache changes the decoder's input signature between the first step and every later step. At step 1 there is no cache yet, so the decoder takes fresh encoder outputs and produces a cache. At step 2 and onward the decoder takes an existing cache plus the previous token and produces an updated cache. ONNX graphs have fixed input and output lists. You cannot declare a graph that sometimes has 2 inputs and sometimes has 74. That is why a single PyTorch decoder becomes two ONNX graphs, and the encoder becomes a third.

The split

The export script 01_export_encoder_decoder.py produces exactly three ONNX files. Each has a distinct job and a distinct input and output contract.

File Runs when Past KV in Present KV out
encoder_model.onnx Once per sentence None None
decoder_model.onnx Step 1 only None Full cache
decoder_with_past_model.onnx Steps 2 through end Full cache Updated cache

A common question is why the encoder is split at all, rather than fused into the first decoder step. Splitting lets the encoder run once and feed all decoder steps. It also matches the I/O layout of the reference naklitechie bundle, so the output works with existing loaders without custom glue.

Why does the decoder become two separate ONNX graphs instead of one?
ONNX cannot represent attention layers in a single graph.
The decoder's input list changes between step 1 (no cache) and step 2+ (cache in), and ONNX graphs have a fixed input and output contract.
Two graphs run faster on WebGPU than one graph.
Correct. The KV cache is absent on step 1 and present on every later step. Because an ONNX graph declares its inputs once at export time, the two signatures require two graphs.
Not quite. The reason is the input contract, not performance or operator support. Think about what inputs the decoder needs on step 1 versus step 5.
Tip: ONNX graphs declare a fixed list of named inputs at export time. They cannot grow that list at runtime.

The encoder graph

The encoder graph is the simplest of the three. It takes the source token ids and an attention mask, and returns the contextual embedding for every source token. It runs once per sentence. Its output, last_hidden_state, is what the decoder attends to through cross attention.

The export wraps the encoder in a thin PyTorch module so the exported graph exposes a clean two input, one output contract.

it2_onnx_wrappers.py
class IndicTransEncoderWrapper(nn.Module):
    def __init__(self, encoder: nn.Module) -> None:
        super().__init__()
        self.encoder = encoder

    def forward(self, input_ids, attention_mask):
        return self.encoder(
            input_ids=input_ids,
            attention_mask=attention_mask,
        ).last_hidden_state

The export call declares two dynamic axes. batch_size lets the graph accept any batch size at runtime. encoder_sequence_length lets it accept sentences of any length up to the model maximum of 256. Without these declarations the traced graph would lock to the dummy shape used during tracing, batch 1 and length 8, and fail on real inputs.

01_export_encoder_decoder.py
torch.onnx.export(
    wrapper,
    (input_ids, attention_mask),
    str(path),
    input_names=["input_ids", "attention_mask"],
    output_names=["last_hidden_state"],
    dynamic_axes={
        "input_ids": {0: "batch_size", 1: "encoder_sequence_length"},
        "attention_mask": {0: "batch_size", 1: "encoder_sequence_length"},
        "last_hidden_state": {0: "batch_size", 1: "encoder_sequence_length"},
    },
    opset_version=17,
    do_constant_folding=True,
    dynamo=False,
)

do_constant_folding=True tells the exporter to precompute parts of the graph that depend only on constants, folding them into stored initializers. dynamo=False forces the legacy TorchScript tracer instead of the newer dynamo exporter. The dynamo path requires an extra dependency called onnxscript and was less reliable for this custom architecture, so the pipeline pins the legacy tracer.

The decoder (first step) graph

decoder_model.onnx handles the first generation step. It receives the start token and the full encoder output, and it produces two things: the logits used to pick the first real token, and the complete initial KV cache that every later step will carry.

Its inputs are input_ids (the single start token), encoder_attention_mask, and encoder_hidden_states (the encoder output tensor). Notice the encoder output is a direct input here. On step 1 the cache is empty, so cross attention must read the fresh encoder states.

Its outputs are logits plus one set of four present KV tensors per decoder layer. With 18 layers that is 72 cache tensors leaving the graph. The wrapper flattens the nested cache into a flat tuple of named tensors so ONNX can declare each one explicitly.

it2_onnx_wrappers.py
class IndicTransDecoderWrapper(nn.Module):
    """First decode step: no past KV in, full present KV out + logits."""

    def forward(self, input_ids, encoder_attention_mask, encoder_hidden_states):
        out = self.decoder(
            input_ids=input_ids,
            attention_mask=None,
            encoder_hidden_states=encoder_hidden_states,
            encoder_attention_mask=encoder_attention_mask,
            use_cache=True,
        )
        logits = self.lm_head(out.last_hidden_state)
        return (logits, *_flatten_past(out.past_key_values))

The output names follow a strict convention. For each layer index i the graph emits present.i.decoder.key, present.i.decoder.value, present.i.encoder.key, and present.i.encoder.value. The decoder self attention cache and the cross attention cache are emitted together. This naming is what the runtime loop uses to feed the next graph.

The decoder_with_past graph

decoder_with_past_model.onnx is the workhorse. It runs for every step after the first. It receives the previous token, the encoder attention mask, and the full cache from the prior step. It returns updated logits and an updated cache with one more entry appended to the decoder self attention portion.

Two non-obvious tricks live in this wrapper. Both fix subtle tracing bugs that produced broken translations in earlier exports.

The dummy encoder hidden states trick

The AI4Bharat modeling code guards cross attention with a check: if encoder_hidden_states is not None, run cross attention. During step 2 and later, the real encoder information already lives in the cached cross attention keys and values. So you might pass None for encoder_hidden_states to signal "use the cache".

That breaks the export. When the tracer sees None, it skips recording the cross attention block entirely. The exported graph then has no cross attention operators at all, so it produces repetitive garbage from step 2 onward. The fix is to pass a dummy zero tensor shaped like the encoder output. The presence of a real tensor makes the tracer compile the cross attention block. At runtime the cross attention reads from the cached keys and values, and the dummy states are projected away.

The zero-cost mask dependency

The ONNX optimizer noticed that encoder_attention_mask appeared unused inside the wrapper's forward path, because the cached cross attention already incorporated the mask. The optimizer then dropped the mask from the graph's input list. At runtime the session rejected calls that supplied the mask, or silently misused it.

The fix is a single line that creates a zero-cost dependency on the mask.

it2_onnx_wrappers.py
# Keep encoder_attention_mask in the ONNX graph (cross-attn mask for cached KV)
logits = logits + encoder_attention_mask.sum() * 0.0

Multiplying the mask sum by zero adds nothing to the logits numerically. But the tracer now records a data flow edge from encoder_attention_mask into the output. The optimizer keeps the input. The mask stays in the graph's public contract without changing any result.

Why is a dummy zero tensor passed as encoder_hidden_states in the decoder_with_past wrapper?
The decoder needs fresh encoder context at every step.
Passing None makes the tracer skip the cross attention block, so a dummy tensor forces that block to compile while the real context is read from the cache at runtime.
Zeros are required for numerical stability of attention.
Correct. The dummy exists to satisfy the tracer. The cached cross attention keys and values already hold the encoder context for steps 2 and onward.
Not quite. The decoder does not need fresh encoder context every step. Reconsider what the tracer does when it encounters a None input.
Tip: Tracing records only the code paths actually executed with the example inputs. A None input can hide a whole branch from the exported graph.

The dynamic axes for this graph are the most involved. The decoder cache tensors declare a past_decoder_sequence_length dynamic axis that grows by one each step. The present outputs declare a matching past_decoder_sequence_length_plus_1 axis. The encoder cache tensors declare encoder_sequence_length dynamic, so the graph is not locked to the length 8 used during tracing.

The KV cache: four tensors per layer

Every decoder layer contributes four cache tensors. Two belong to self attention and grow each step. Two belong to cross attention and stay fixed at the encoder sequence length. Understanding this split is the key to reading any seq2seq ONNX export.

Tensor Source Shape (batch, heads, seq, head_dim) Grows per step?
decoder.key Self attention B, 8, T, 64 Yes, T increases by 1
decoder.value Self attention B, 8, T, 64 Yes, T increases by 1
encoder.key Cross attention B, 8, S, 64 No, fixed at encoder length S
encoder.value Cross attention B, 8, S, 64 No, fixed at encoder length S

With 18 layers, each step carries 72 cache tensors in and emits 72 out. The encoder cross attention half of the cache is set once at step 1 and then carried unchanged through every later step. Only the decoder self attention half grows. This is why step 1 is expensive, it computes the full cross attention cache from the encoder output, and later steps are cheap, they only extend the self attention cache by one entry.

External data sidecars

An ONNX protobuf has a hard practical size limit around 2 GB because of the 32 bit message size cap in the protobuf format. The decoder weights for the 200M model approach or exceed that limit. The export pipeline handles this with a sidecar file.

The function _externalize_if_large checks the serialized proto size. If it exceeds 512 MB, it calls convert_model_to_external_data to move every weight tensor into a separate binary file named encoder_model.onnx.data or decoder_model.onnx.data. The proto then keeps only the graph structure and small metadata, while the bulk weights live in the sidecar.

01_export_encoder_decoder.py
def _externalize_if_large(onnx_path: Path, size_threshold_mb=512) -> None:
    model = onnx.load(str(onnx_path), load_external_data=True)
    proto_mb = onnx_path.stat().st_size / (1024 * 1024)
    if proto_mb < size_threshold_mb:
        return
    convert_model_to_external_data(
        model,
        all_tensors_to_one_file=True,
        location=data_path.name,
        size_threshold=1024,
    )
    save_model(model, str(onnx_path))

In the real en to indic bundle this produces a striking split. The encoder proto is 294 MB, under the threshold, so its weights stay inside encoder_model.onnx. Each decoder proto is about 1.2 MB of graph structure, while the actual weights live in a 805 MB and 767 MB sidecar respectively. You must ship the .onnx.data file next to its .onnx file. The runtime loads both automatically as long as they sit in the same directory.

If you upload a bundle to Hugging Face without the .onnx.data sidecars, the runtime will fail to load the decoders. Treat each .onnx plus its .onnx.data as an inseparable pair.

Tokenizers: the other half of the bundle

The ONNX graphs only know about integer token ids. Turning raw text into ids and ids back into text is the job of the tokenizer, which ships as a separate set of files in the same bundle. For IndicTrans2 this is where most of the export difficulty lives.

Slow versus fast

The original model ships a slow tokenizer implemented in Python, the file tokenization_indictrans.py, loaded through trust_remote_code=True. It wraps two SentencePiece models, model.SRC and model.TGT, plus two Fairseq dictionaries, dict.SRC.json and dict.TGT.json. The slow tokenizer works in Python but cannot run efficiently in a browser.

The browser needs a fast tokenizer, the single file format tokenizer.json understood by the Rust tokenizers library and by onnxruntime-web's tokenizer support. The build script 02_build_fast_tokenizers.py converts the SentencePiece models into tokenizer_src.json and tokenizer_tgt.json using the Hugging Face SpmConverter.

Why the dict remap is necessary

The conversion is not a one step copy. SpmConverter assigns token ids using SentencePiece's native indexing, which does not match the Fairseq dictionary ids the model was trained on. If you shipped the raw converted file, the model would receive the wrong id for almost every token and produce garbage.

The build script remaps every vocabulary entry to the id from the matching dict json. It then registers the special tokens and the language tags as added tokens with their correct dict ids. The language tags like hin_Deva and eng_Latn are marked single_word: true and special: false, matching the reference bundle. Finally it installs a TemplateProcessing post-processor that appends the end of sentence token with id 2.

02_build_fast_tokenizers.py
remapped = {
    token: vocab_dict.get(token, UNK_ID)
    for token in fast["model"]["vocab"]
}
fast["model"]["vocab"] = remapped
fast["added_tokens"] = added_tokens
fast["post_processor"] = EOS_POST_PROCESSOR

A small tokenizer_meta.json file records src_dict_size, tgt_dict_size, and unk_id: 3. The runtime uses these to clamp any id greater than or equal to the dict size down to the unknown token. This guards against the fast tokenizer emitting ids for vocabulary entries the model never trained on.

Encoding a source sentence also requires a language prefix. The runtime prepends the source and target language tags before the text, for example hin_Deva eng_Latn यह एक परीक्षण वाक्य है।. The added tokens setup ensures the tokenizer recognizes those tags and maps them to their dict ids.

Why two tokenizers

Source and target use separate vocabularies, separate SentencePiece models, and separate dictionaries. The source side encodes input text in the source script. The target side decodes generated ids back to the target script. They are not interchangeable. Swapping a source tokenizer from one direction into another direction gives zero percent token match, because the dictionaries interleave language tags at different ids. Each direction builds its own pair from scratch.

After SpmConverter produces a fast tokenizer, why is an extra remap step required?
The fast format is missing the post-processor.
SpmConverter uses SentencePiece-native ids that differ from the Fairseq dict ids the model was trained on, so ids must be remapped to the dict json values.
The conversion drops language tags, so they must be reinserted manually.
Correct. Without the remap the model receives ids from the wrong vocabulary indexing and produces garbage output.
Not quite. The post-processor and language tags are added separately. The remap exists because of an id numbering mismatch between two vocabulary systems.
Tip: SentencePiece and Fairseq are two different tokenization systems with their own id assignments. The model was trained against the Fairseq ids.

Config files

Two json files carry model level configuration into the bundle. config.json holds the architecture constants, layer counts, head counts, embedding dimensions, vocabulary sizes, and token ids. The runtime does not always need it to run the graphs, but loaders and validation scripts read it to recover the layer count and dimensions.

generation_config.json holds the decoding parameters. The two values that matter most for the greedy loop are decoder_start_token_id and eos_token_id, both 2 for these models. The runtime uses the start id to seed the decoder and the eos id to detect when generation is complete. Without generation_config.json the validation script falls back to defaults, which happen to match, but shipping the file makes the bundle self describing.

The remaining tokenizer support files, tokenizer_config.json, special_tokens_map.json, and tokenization_indictrans.py, are copied from the original repo. They let the slow tokenizer still load in Python for validation and postprocessing even though the browser uses the fast json tokenizers.

Putting it together: the greedy decode loop

The validation script 03_validate_parity.py contains the cleanest statement of how all the components cooperate. It runs the same greedy loop against both the PyTorch model and the three ONNX sessions and compares the emitted token ids. The pass bar is 99 percent token exact match. All three directions hit 100 percent on their fixtures.

The loop is short. The encoder session runs once. The decoder session runs for step 1. The decoder_with_past session runs for every step after that, carrying the cache forward each time. Argmax over the last logits position picks the next token. When that token equals the eos id, the loop stops.

03_validate_parity.py
enc_out = enc.run(["last_hidden_state"], {
    "input_ids": input_ids,
    "attention_mask": attn_mask,
})[0]

decoder_input_ids = np.array([[decoder_start_id]], dtype=np.int64)
output_ids = [decoder_start_id]
past_outputs = None

for step in range(max_new_tokens):
    if step == 0:
        dec_out = dec.run(None, {
            "input_ids": decoder_input_ids,
            "encoder_hidden_states": enc_out,
            "encoder_attention_mask": attn_mask,
        })
    else:
        dec_out = dec_past.run(None, {
            "input_ids": decoder_input_ids,
            "encoder_attention_mask": attn_mask,
            **_past_feed_from_outputs(past_outputs, num_layers),
        })

    logits = dec_out[0]
    past_outputs = list(dec_out[1:])
    next_id = int(np.argmax(logits[0, -1, :]))
    output_ids.append(next_id)
    if next_id == eos_id:
        break
    decoder_input_ids = np.array([[next_id]], dtype=np.int64)

The helper _past_feed_from_outputs is the bridge between the two decoder graphs. It takes the flat list of present outputs from the previous step and renames them into the past_key_values.* input names the next step expects. The encoder cross attention half of that feed never changes after step 1. Only the decoder self attention half grows.

Note that model.generate(), the usual Hugging Face generation entry point, does not work on this custom architecture. It raises an attribute error around use_cache. The manual loop is the workaround, and it doubles as the reference implementation that the browser worker later mirrors in TypeScript.

Quantization: a second tier

The fp32 bundles are large. The en to indic bundle is about 1.7 GB. For low bandwidth or mobile deployments, the pipeline offers an optional INT8 tier built by 04_quantize_int8.py. It runs dynamic quantization on the weights of all three graphs, leaving activations in floating point.

Dynamic quantization replaces weight matrices with int8 lookups and small scale and zero point metadata. The script uses QuantType.QInt8 and per channel scaling, which keeps a separate scale per output channel and usually preserves more quality than per tensor scaling. The tokenizer and config files are copied across unchanged from the fp32 bundle, because quantization only touches the graph weights.

04_quantize_int8.py
quantize_dynamic(
    model_input=str(src),
    model_output=str(dst),
    weight_type=QuantType.QInt8,
    per_channel=True,
)

The tradeoff is real. The reference int8 benchmark hits about 80 percent exact text match, well below the 99 percent bar that fp32 clears. INT8 ships as a separate preview repo with a warning that it is not production quality. The roadmap also lists fp16 and q4f16 as future tiers with better size to quality tradeoffs.

How are these values calculated?

The size estimate starts from the measured fp32 en to indic bundle size of 1.7 GB and applies a reduction ratio per tier. fp16 halves the weight bytes, giving roughly 0.5 times the fp32 size. INT8 reduces 32 bit weights to 8 bit, giving roughly 0.25 times the fp32 size. q4f16 uses 4 bit weights with 16 bit activations, giving roughly 0.125 times the fp32 size. A fixed overhead of 90 MB is added to every tier to account for the two fast tokenizer json files and the config files, which are not quantized.

size model
FP32_BASE_GB = 1.7
OVERHEAD_GB   = 0.09
RATIO = {"fp32": 1.0, "fp16": 0.5, "int8": 0.25, "q4f16": 0.125}
size_gb = max(FP32_BASE_GB * RATIO[tier] * 0.94 + OVERHEAD_GB, OVERHEAD_GB)

The 0.94 factor accounts for the roughly 6 percent of the fp32 bundle that is already non weight data, tokenizer json and configs, which the ratio should not shrink. Parity values come from the measured fp32 result of 100 percent and the documented int8 benchmark of about 80 percent. fp16 and q4f16 parity is projected, not yet measured, and marked as such.

These numbers assume the en to indic 200M model specifically. The indic to indic 320M model scales up by roughly the parameter ratio, and q4f16 quality for Indic scripts is still unverified.

Every file above serves one of three roles. The three .onnx graphs plus their .data sidecars hold the model weights and operator graph. The tokenizer files turn text into ids and back.
